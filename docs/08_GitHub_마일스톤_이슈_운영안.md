# GitHub Milestone · Issue 운영안

## 문서 목적

이 문서는 Skinny 프로젝트를 GitHub의 `Milestone`과 `Issue`로 운영하기 위한 기준안입니다.

프로젝트 특성상 도메인 중심으로 구조를 나누고 있지만,
실제 개발 과정에서는 설계와 책임 경계가 바뀔 수 있습니다.
그래서 이 문서는 아래 두 가지를 동시에 만족하도록 구성합니다.

- Milestone은 사용자가 체감하는 기능 가치 단위로 묶는다
- Issue는 하루~이틀 안에 끝낼 수 있는 유연한 작업 단위로 나눈다

---

## 프로젝트 개요

Skinny는 화장품 성분을 분석하고, 내 피부에 더 안전한 제품을 찾도록 돕는 백엔드 프로젝트입니다.

핵심 흐름은 다음과 같습니다.

- 화장품 라벨 이미지를 OCR로 읽는다
- 성분을 표준 INCI 기준으로 정규화한다
- Validation Engine으로 위험 조합과 개인 기피 성분을 확인한다
- Knowledge Base 기반 RAG로 성분 설명을 만든다
- 사용자 피부 프로파일과 트러블 로그를 바탕으로 개인화 추천을 제공한다

핵심 도메인은 다음과 같습니다.

- `Ingredient`
- `Validation`
- `OCR / Scan`
- `RAG`
- `UserProfile`
- `TroubleLog`
- `Recommendation`

## 운영 원칙

### 1. Milestone 기준

- Milestone은 기술 레이어 기준이 아니라 `사용자에게 전달되는 가치` 기준으로 나눈다
- 너무 잘게 나누지 않고 `3~5개` 수준으로 유지한다
- 구조 변경이 생겨도 Milestone 이름은 최대한 유지하고, 내부 Issue만 조정한다

### 2. Issue 기준

- Issue는 `하루 ~ 이틀 안에 끝낼 수 있는 크기`로 자른다
- 하나의 Issue는 하나의 책임만 가진다
- domain 기준으로 제목을 시작한다
- 완료 조건이 불명확한 작업은 Issue로 만들지 않는다

### 3. 변경 대응 기준

- 설계가 바뀌면 기존 Issue를 억지로 유지하지 않는다
- 하나의 Issue가 커지면 분리하고, 너무 작은 Issue는 합친다
- 이미 구현된 작업도 안정화/정책 정리가 필요하면 별도 Issue로 다시 만든다

### 4. Commit vs Issue 구분 기준

GitHub Issue는 `작업 단위`, Commit은 `구현 단위`로 사용한다.

하나의 Issue는 여러 개의 Commit으로 구성될 수 있으며,
Commit은 해당 Issue를 완성하기 위한 코드 변경 기록이다.

#### 4.1 Commit으로 처리하는 경우

다음과 같은 변경은 새로운 Issue를 만들지 않고,
기존 Issue 내부에서 Commit으로 처리한다.

- 구현 중 발생한 자연스러운 로직 수정
- 단순 버그 수정
- 변수명 또는 함수명 변경
- 리팩토링 수준의 구조 개선
- 테스트 코드 추가 및 수정

즉, `작업의 목적이 변하지 않는 변경`은 Commit으로 처리한다.

#### 4.2 새로운 Issue를 생성해야 하는 경우

다음과 같은 경우는 반드시 새로운 Issue로 분리한다.

- 기능이 확장되는 경우
- 설계 구조가 변경되는 경우
- 새로운 도메인 개념이 등장하는 경우
- 기존 작업의 목적이 달라지는 경우

예시:

- 단순 검증에서 설명 포함 검증으로 확장되는 경우
- 동기 처리에서 비동기 처리로 구조가 바뀌는 경우
- `unmapped ingredient`, `fallback strategy` 같은 새로운 개념이 등장하는 경우
- 단순 조회 API에서 사용자 UX 개선 중심 API로 목적이 바뀌는 경우

즉, `작업의 목적이 바뀌는 변경`은 새로운 Issue로 분리한다.

#### 4.3 판단 기준

다음 질문으로 판단한다.

`이 변경은 나중에 봤을 때 왜 이렇게 바뀌었는지 설명이 필요한가?`

- YES: 새로운 Issue 생성
- NO: 기존 Issue 내부에서 Commit으로 처리

#### 4.4 Issue와 Commit의 관계

- Issue는 `무엇을 만들 것인가`를 정의한다
- Commit은 `어떻게 만들었는가`를 기록한다

예시:

Issue:

- `[Validation] 성분 조합 검증 API 노출`

Commit:

- repository 조회 로직 추가
- service 검증 로직 구현
- API endpoint 연결
- 테스트 코드 작성

## Milestone 구성

| Milestone | 사용자 가치 | 포함 도메인 |
|------|------|------|
| `M1. 성분 안전 확인 경험` | 성분을 찾고, 조합 위험을 바로 확인할 수 있다 | Ingredient, Validation |
| `M2. 라벨 스캔 경험` | 라벨 이미지를 올려 성분 인식과 검증 결과를 한 번에 받는다 | OCR, Scan, Ingredient |
| `M3. 설명 포함 통합 조회 경험` | 성분을 찾고, 검증하고, 근거 기반 설명까지 한 흐름으로 확인한다 | Ingredient, Validation, RAG |
| `M4. 개인화 추천 경험` | 내 피부 타입과 트러블 이력에 맞는 추천을 받는다 | UserProfile, TroubleLog, Recommendation |

## M1. 성분 안전 확인 경험

### Milestone 설명

이 Milestone의 목적은 사용자가 성분 조합의 위험을 신뢰할 수 있도록,
성분 데이터와 검증 규칙의 기준을 먼저 안정화하는 것입니다.

여기서는 조회 API를 늘리는 것보다
`alias`, `충돌 데이터`, `severity`, `응답 해석 정책`을 고정하는 데 집중합니다.

### Issue

#### [Ingredient] 성분 alias 기준 정리

## 목적
검색, 스캔, 검증 흐름에서 같은 성분이 다른 이름으로 흔들리지 않도록
alias와 표준 성분 연결 기준을 고정한다.

## 작업 내용
- INCI 이름, alias, 한글명 간 매핑 기준 정리
- 중복 alias와 충돌 alias 처리 규칙 점검
- 검색/스캔/검증에서 공통으로 쓸 alias 조회 기준 정리
- alias 매핑 관련 테스트 보강

## 완료 조건
- 같은 성분이 여러 입력명으로 들어와도 일관되게 정규화된다
- alias 매핑 규칙이 테스트로 고정된다

#### [Validation] 충돌 데이터와 조회 규칙 정리

## 목적
성분 충돌이 명시적 데이터 기반으로 일관되게 탐지되도록
충돌 pair 저장 기준과 조회 규칙을 고정한다.

## 작업 내용
- 충돌 pair의 정렬 저장 기준 점검
- seed 데이터 포맷과 필수 필드 정리
- 입력 순서와 무관하게 같은 pair가 조회되는지 테스트 보강
- severity와 reason 누락 케이스 점검

## 완료 조건
- 충돌 데이터의 저장 기준이 문서와 코드에서 일치한다
- 입력 순서가 바뀌어도 같은 충돌이 탐지된다

#### [Validation] 피부 타입 규칙 severity 정책 정리

## 목적
피부 타입 규칙이 단순 경고인지 실제 차단인지 기준을 명확히 해,
Validation과 Recommendation에서 같은 의미로 해석되도록 만든다.

## 작업 내용
- `SKIN_TYPE_RULES`의 severity 의미 문서화
- LOW / MID / HIGH가 warning인지 blocker인지 기준 정리
- `ValidationService`, `RecommendationService`의 해석 차이 확인
- 관련 회귀 테스트 추가

## 완료 조건
- 피부 타입 규칙 severity의 해석 기준이 정리된다
- validation과 recommendation에서 서로 다른 해석이 남지 않는다
- LOW 규칙이 과도하게 차단되지 않는다

#### [Validation] 검증 응답 정책 정리

## 목적
검증 결과를 프론트가 예측 가능하게 사용할 수 있도록 응답 의미를 명확히 한다.

## 작업 내용
- `is_safe`, `severity`, `conflicts`, `personal_warnings`의 의미 정리
- layer별 warning이 severity에 어떻게 반영되는지 규칙 점검
- 예시 응답과 테스트 케이스 정리

## 완료 조건
- 응답 필드의 의미가 문서와 코드에서 일치한다
- 프론트가 분기 가능한 수준으로 응답이 고정된다

## M2. 라벨 스캔 경험

### Milestone 설명

이 Milestone의 목적은 사용자가 라벨 이미지를 업로드했을 때,
OCR 결과를 그대로 보여주는 수준이 아니라 `정규화된 성분 + 검증 결과`까지 한 번에 받게 하는 것입니다.

핵심은 OCR 정확도보다도 `후처리`, `alias 매핑`, `미매핑 처리`를 안정화하는 데 있습니다.

### Issue

#### [OCR] OCR provider adapter 안정화

## 목적
로컬 개발 환경과 실제 OCR provider 사이의 차이를 흡수하고,
오류가 발생해도 디버깅 가능한 형태로 다룰 수 있게 한다.

## 작업 내용
- OCR provider 선택 방식 정리
- timeout 및 provider 예외 처리 보강
- 개발용 fallback과 운영용 provider 책임 구분
- OCR client 단위 테스트 보강

## 완료 조건
- provider 호출 실패 시 일관된 예외가 발생한다
- 로컬과 운영 환경에서 같은 인터페이스로 OCR을 호출할 수 있다

#### [Scan] OCR 전처리 규칙 보강

## 목적
OCR 원문 품질이 들쭉날쭉해도 성분 토큰 분리가 안정적으로 되도록 만든다.

## 작업 내용
- 줄바꿈, 헤더, 괄호, 숫자, 특수문자 제거 규칙 보강
- 쉼표/세미콜론 기준 분리 로직 점검
- 실이미지 기준 오인식 케이스를 테스트 데이터로 추가

## 완료 조건
- 대표적인 라벨 텍스트에서 토큰 분리가 안정적으로 동작한다
- 불필요한 노이즈 토큰이 줄어든다

#### [Scan] alias 매핑 보강

## 목적
OCR로 읽힌 성분명이 실제 표준 성분으로 더 잘 연결되도록 alias 매핑 품질을 높인다.

## 작업 내용
- 자주 나오는 한글명/변형 표기 alias 추가
- alias 우선순위와 dedupe 기준 점검
- 매핑률 개선 전후를 확인할 테스트 추가

## 완료 조건
- 대표 샘플에서 recognized ingredient 수가 증가한다
- 이미 매핑되던 성분이 회귀하지 않는다

#### [Scan] 미매핑 성분 처리 흐름 정리

## 목적
매핑되지 않은 성분이 있어도 사용자와 프론트가 상황을 이해할 수 있도록 응답을 정리한다.

## 작업 내용
- unmapped 성분 표시 정책 정리
- 저신뢰 OCR fallback과 미매핑 성분 처리 역할 구분
- 미매핑 성분 관련 응답 예시와 테스트 추가

## 완료 조건
- unmapped 성분이 응답에서 명확히 구분된다
- OCR 저신뢰 fallback과 미매핑 케이스가 혼동되지 않는다

#### [Scan] 스캔 결과 조회 API 추가

## 목적
한 번 수행한 스캔 결과를 다시 조회하거나 디버깅할 수 있게 한다.

## 작업 내용
- `scan_results` 조회용 repository/service/API 추가
- raw OCR text, parsed ingredients, validation 결과 재구성 방식 정리
- 스캔 결과 조회 API 테스트 추가

## 완료 조건
- 저장된 스캔 결과를 ID 기준으로 다시 조회할 수 있다
- 원본 OCR 텍스트와 정규화 결과를 함께 확인할 수 있다

## M3. 설명 포함 통합 조회 경험

### Milestone 설명

이 Milestone의 목적은 사용자가 성분을 찾고,
바로 검증하고, 필요하면 근거 기반 설명까지 이어서 확인할 수 있는
`통합 조회 흐름`을 만드는 것입니다.

따라서 이 Milestone은 RAG만 따로 떼어 구현하는 것이 아니라,
Ingredient API, Validation API, explanation 흐름이 하나의 사용자 경험으로 이어지도록 정리합니다.

### Issue

#### [Ingredient] 성분 조회·검색 API 연결

## 목적
성분 도메인을 내부 로직이 아니라 실제 API 리소스로 사용할 수 있게 만든다.

## 작업 내용
- 성분 상세 조회 API 구현
- 성분 검색 API 구현
- alias 검색 결과를 응답에 포함하도록 정리
- `IngredientService`와 repository 연결
- API 테스트 추가

## 완료 조건
- 성분 ID로 상세 조회가 가능하다
- 검색어로 성분 검색이 가능하다
- alias 기반 검색이 동작한다

#### [Validation] 성분 조합 검증 API 노출

## 목적
스캔 흐름 밖에서도 성분 조합 검증을 독립적으로 사용할 수 있게 한다.

## 작업 내용
- 성분 이름 기반 검증 API 연결
- 성분 ID 기반 검증 API 연결
- `ValidationService` 응답을 API 스키마로 정리
- 검증 API 테스트 추가

## 완료 조건
- 이름 목록으로 검증 요청이 가능하다
- ID 목록으로 검증 요청이 가능하다
- conflicts, severity, personal_warnings가 일관되게 반환된다

#### [RAG] KB 검색 + guardrail 구현

## 목적
Knowledge Base를 기반으로 성분 설명이 가능한지 먼저 판단하고,
근거가 없을 때는 생성 자체를 차단한다.

## 작업 내용
- 로컬 KB 검색 흐름 구현
- 검색 결과 DTO와 source 메타데이터 구조 정리
- 검색 결과 empty 체크와 no-result guardrail 추가
- 생성 차단 테스트 추가

## 완료 조건
- KB가 있는 성분은 관련 문서를 찾을 수 있다
- KB가 없는 성분은 설명 생성을 시도하지 않는다
- no-result guardrail 테스트가 통과한다

#### [RAG] 설명 생성 클라이언트 연결

## 목적
검색 결과를 바탕으로 설명을 생성하고,
이를 API 응답으로 연결할 수 있는 생성 경로를 완성한다.

## 작업 내용
- LLM client 인터페이스 정의
- prompt에 검색된 근거만 쓰도록 제한
- timeout/empty response 예외 처리 추가
- `GET /ingredients/{id}/explain` 응답 스키마와 API 연결
- 설명 API 테스트 추가

## 완료 조건
- 검색 결과를 바탕으로 grounded 설명을 생성할 수 있다
- 설명 API가 summary와 source 목록을 함께 반환한다
- provider 실패 또는 근거 없음 상황에서 안전한 응답을 반환한다

## M4. 개인화 추천 경험

### Milestone 설명

이 Milestone의 목적은 추천 기능을 "단순 목록 반환" 수준이 아니라,
`안전 필터 → 개인화 필터 → 점수화 → 설명` 흐름으로 완성하는 것입니다.

여기서는 추천 엔진 전체를 한 번에 만드는 대신,
단계별로 책임을 분리해 Issue를 운영합니다.

### Issue

#### [UserProfile] 프로파일 입력 정규화 정리

## 목적
피부 타입, 피부 고민, 기피 성분 입력이 추천과 검증에 일관되게 반영되도록 만든다.

## 작업 내용
- skin concern normalization 기준 정리
- 프로파일 필드 validation 보강
- 프로파일 변경 시 추천 캐시 무효화 조건 점검

## 완료 조건
- 프로파일 입력값이 예측 가능한 형태로 저장된다
- 추천 캐시가 잘못 남지 않는다

#### [TroubleLog] 반복 성분 제안 흐름 보강

## 목적
트러블 로그가 단순 기록에 그치지 않고, 개인화 필터의 입력으로 자연스럽게 연결되도록 한다.

## 작업 내용
- 반복 등장 성분 집계 기준 점검
- suggestion 생성 기준 정리
- 제안 응답 형식 보강
- 관련 서비스 테스트 추가

## 완료 조건
- 반복된 트러블 성분이 제안 목록으로 안정적으로 반환된다
- 제안 기준이 테스트로 고정된다

#### [TroubleLog] confirm 후 기피 성분 등록 정책 정리

## 목적
자동 등록 제안과 실제 저장 시점을 분리해 사용자 제어권을 유지한다.

## 작업 내용
- confirm 전/후 상태 분리
- 이미 등록된 기피 성분 처리 기준 정리
- confirm API 예외 처리 보강

## 완료 조건
- 사용자 confirm 없이 기피 성분이 저장되지 않는다
- confirm 이후에만 avoid ingredient가 반영된다

#### [Recommendation] 후보 필터 기준 정리

## 목적
추천 후보 제거 기준을 명확히 해, 과도한 차단이나 누락이 생기지 않도록 만든다.

## 작업 내용
- high conflict, skin type blocker, avoid ingredient, trouble history의 필터 정책 정리
- validation 단계와 개인화 필터 단계의 책임 구분
- 회귀 테스트 추가

## 완료 조건
- 어떤 조건에서 후보가 제외되는지 설명 가능하다
- 필터 정책이 테스트로 고정된다

#### [Recommendation] 점수 계산 규칙 분리

## 목적
피부 타입, 피부 고민, 트러블 이력 가중치를 추천 필터와 분리해 조정 가능하게 만든다.

## 작업 내용
- 점수 계산 팩터를 함수 단위로 분리
- 가중치 상수 정리
- 대표 케이스에 대한 점수 테스트 추가

## 완료 조건
- 점수 계산 로직이 독립적으로 읽히고 조정 가능하다
- 점수화 회귀가 테스트로 잡힌다

#### [Recommendation] 추천 이유 생성 연결

## 목적
추천 결과에 "왜 이 제품이 추천되는지"를 자연어로 설명할 수 있게 한다.

## 작업 내용
- 상위 추천 결과와 RAGService 연결
- 추천 이유 생성용 입력 구조 정의
- RAG 실패 시 reason 없이 반환하는 fallback 정리

## 완료 조건
- 추천 결과에 reason 필드가 채워진다
- RAG 실패 시 추천 목록은 유지된다

#### [Recommendation] 추천 0건 fallback 정책 정리

## 목적
조건이 너무 강해 추천 결과가 없을 때 사용자에게 다음 행동을 안내할 수 있게 한다.

## 작업 내용
- 0건 판단 기준 정리
- 필터 완화 제안 메시지 작성
- fallback 응답 스키마와 테스트 정리

## 완료 조건
- 추천 결과가 없을 때 빈 배열만 주지 않는다
- 사용자가 다음 행동을 이해할 수 있는 메시지를 받는다

## 실제 GitHub 적용 순서 제안

처음부터 모든 Issue를 한 번에 만들 필요는 없습니다.
아래 순서로 시작하는 것이 가장 부담이 적습니다.

1. Milestone 4개만 먼저 생성한다
2. 현재 바로 작업할 Milestone 1개를 고른다
3. 그 Milestone 아래에서 오늘~내일 처리할 Issue 2~3개만 먼저 만든다
4. 작업하면서 나머지 Issue를 추가로 생성한다

예시:

- 이번 주에 성분 조회와 설명 흐름을 붙일 계획이면 `M3. 설명 포함 통합 조회 경험`만 먼저 연다
- 그 아래에서
  - `[Ingredient] 성분 조회·검색 API 연결`
  - `[Validation] 성분 조합 검증 API 노출`
  - `[RAG] KB 검색 + guardrail 구현`
  - `[RAG] 설명 생성 클라이언트 연결`
  네 개만 먼저 생성해도 충분하다

## 한 줄 정리

Milestone은 `사용자 가치 단위`, Issue는 `하루~이틀 안에 끝낼 수 있는 도메인 단위`로 운영한다.
설계가 바뀌면 문서를 고치기보다 Issue를 나누고 합치면서 유연하게 관리한다.
