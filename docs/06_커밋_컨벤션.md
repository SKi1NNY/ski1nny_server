# Git 커밋 메시지 컨벤션

본 프로젝트는 **Conventional Commits** 형식을 기반으로 커밋 메시지를 작성합니다.

---

## 1. 기본 포맷

```
<type>[optional scope]: <description>

[optional body]

[optional footer(s)]
```

---

## 2. 구성 요소

### 1) Header

```
<type>[optional scope]: <description>
```

예시

```
feat: 사용자 프로필 페이지 추가
fix(auth): 로그인 토큰 검증 오류 수정
```

규칙

* 커밋 제목의 description은 한국어로 작성한다
* type과 scope는 Conventional Commits 형식을 유지한다

---

### type

변경 사항의 성격을 나타냅니다.

| type     | 설명                   |
| -------- | -------------------- |
| feat     | 새로운 기능 추가            |
| fix      | 버그 수정                |
| refactor | 동작 변화 없는 코드 구조 개선    |
| docs     | 문서 변경                |
| chore    | 설정 / 빌드 / 의존성 변경     |
| test     | 테스트 코드 추가 및 수정       |
| style    | 코드 스타일 변경 (로직 변경 없음) |

---

### scope (선택)

변경된 영역을 명시할 때 사용합니다.

예

```
feat(auth): 로그인 기능 추가
fix(api): 사용자 조회 오류 수정
```

---

### description

변경 내용을 간단히 요약합니다.

권장 규칙

* 50자 이내 작성
* 간결하게 작성
* 마침표 사용하지 않음
* description은 한국어로 작성
* `T-01`, `T-87-1` 같은 투두 번호를 직접 쓰지 않음

예

```
feat: 사용자 프로필 페이지 추가
```

---

## 3. Body (필수)

모든 커밋은 Body를 작성합니다.

보통 **bullet list 형태**로 작성합니다.

예

```
- 사용자 프로필 페이지 및 라우팅 구현
- 프로필 정보를 보여주는 프로필 카드 컴포넌트 구현
- 프로필 수정 기능 구현
```

권장 규칙

* 변경 사항을 중심으로 작성
* 한 줄은 72자 이내
* 최소 1개 이상의 bullet line을 작성한다
* Body에도 투두 번호를 직접 쓰지 않는다

---

## 4. Footer (선택)

이슈 번호 또는 참고 정보를 추가합니다.

예

```
Closes #123
Fixes #45
Refs #78
```

---

## 5. Issue와 Commit의 역할 차이

GitHub `Issue`와 `Commit`은 같은 이름으로 관리할 필요가 없습니다.

- Issue는 `무엇을 왜 할 것인가`를 정의한다
- Commit은 `어떻게 구현했는가`를 기록한다

즉, Issue 제목은 작업 목적 중심으로 작성하고,
Commit 제목은 실제 코드 변경 중심으로 작성하는 것이 좋습니다.

예시

Issue

- `[Validation] 성분 조합 검증 API 노출`

Commit

- `feat(validation): 성분 검증 엔드포인트 추가`
- `test(validation): 성분 검증 API 테스트 추가`

권장 규칙

- Issue 제목을 커밋 제목으로 그대로 복사하지 않는다
- 커밋은 endpoint 추가, service 연결, 테스트 보강처럼 구현 중심으로 쓴다
- 하나의 Issue는 여러 Commit으로 나뉠 수 있다
- 하나의 Commit이 여러 관련 Issue를 함께 진전시킬 수도 있다
- 관련 Issue가 있다면 footer에서 `Refs #번호`로 연결한다

---

## 6. 커밋 메시지 예시

```
feat: 사용자 프로필 페이지 추가

- 사용자 프로필 페이지 및 라우팅 구현
- 프로필 정보를 보여주는 프로필 카드 컴포넌트 구현
- 프로필 수정 기능 구현

Closes #123
```

잘못된 예시

```
feat: add user profile page
```

---

## 7. 작성 규칙 요약

* `type: description` 형식을 사용한다
* description은 한국어로 작성한다
* description은 간결하게 작성한다
* body는 반드시 작성하며 bullet list 형태를 사용한다
* Header와 Body에 투두 번호를 직접 쓰지 않는다
* footer에는 이슈 번호를 연결할 수 있다
* Issue는 목적 중심, Commit은 구현 중심으로 구분한다
